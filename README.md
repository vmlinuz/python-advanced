# Python Advanced

Цей репозиторій — центральна робоча база курсу з просунутого Python: пам’ять і внутрішні механізми інтерпретатора,
конкурентність і паралелізм, інженерія продуктивності та архітектурне мислення.

Тут зібрані лекційні матеріали, домашні завдання, автоматичні тести та рішення від лектора. Репозиторій
виступає **єдиним джерелом правди** протягом усього курсу і водночас є вашим основним робочим середовищем.


## Структура репозиторію

- `initial/` стартові модулі та перші домашні завдання;
- `memory/` матеріали та приклади з внутрішньої організації пам’яті;
- `multitasking/` теми про GIL, потоки, процеси та асинхронність;
- `tests/` тести для домашніх завдань, розбиті за модулями;
- `requirements.txt` залежності курсу;
- `pyproject.toml` базова конфігурація інструментів.

## Налаштування середовища

Вимога курсу — Python 3.12+. Рекомендована практика — ізольоване віртуальне середовище на рівні репозиторію.

### 1. Створіть і активуйте віртуальне середовище

```bash
    python -m venv .venv
    source .venv/bin/activate
```

### 2. Встановіть залежності

```bash
    pip install -r requirements.txt
```

## Формат роботи з репозиторієм

Цей курс використовує модель з двома remote, яка відокремлює «джерело правди» від вашої робочої копії.

- `upstream` — репозиторій лектора, read-only
- `origin` — ваш приватний репозиторій, куди ви пушите домашні

### 1. Налаштуйте remotes

```bash
    git remote add upstream <URL_репозиторію_лектора>
    git remote add origin <URL_вашого_репозиторію>
```

### 2. Регулярно отримуйте оновлення курсу

```bash
    git fetch upstream
    git merge upstream/main
```

### 3. Відправляйте свої рішення в `origin`

```bash
    git checkout -b <branch_name>
    git push origin <branch_name>
```

### 4. Pre-commit
У репозиторії вже налаштовано pre-commit хуки — вони автоматично запускаються перед кожним git commit і перевіряють
базову якість коду. Якщо перевірки не проходять — коміт буде відхилено.

Що перевіряє pre-commit
- Ruff — lint і частково автофікси;
- базові git‑хуки (whitespace, eof тощо).

У більшості випадків вам нічого не потрібно робити вручну — просто робіть коміт.

Для налаштування переконайтеся, що pre-commit встановлено:
```bash
  pip install pre-commit
  pre-commit install
```

Якщо треба запустити перевірку вручну:
```bash
  pre-commit run --all-files
```

Якщо треба запускати інструменти напряму:
- Ruff:
```bash
  ruff check .
  ruff check . --fix
  ruff format .
```
- Тести:
```bash
  pytest .
```

## Як працювати з домашніми завданнями

Домашні завдання — це передусім про інженерне мислення, а не про механічне «пройти тести». Правильний порядок роботи
такий: спочатку — розуміння задачі й вибір підходу, далі — реалізація, і лише після цього — використання тестів як
інструменту перевірки.

TDD можна застосовувати як допоміжну стратегію, якщо складно зайти в задачу лише з опису. Водночас ваша здатність
аргументувати технічні рішення значно важливіша за швидкість написання коду.


### З чого складається домашнє завдання
У кожному розділі (initial, memory, multitasking) ви знайдете:
- Презентацію — її необхідно уважно опрацювати, усвідомити матеріал і виписати питання, які залишилися незрозумілими,
щоб обговорити їх на наступній зустрічі;
- Блок «Що почитати?» в кінці кодної презентації — рекомендовані статті та матеріали для поглиблення теми;
- Директорію homework/ — практичні завдання відповідного модуля;
- Директорію homework_answers/ — запропоновані рішення домашніх завдань від лектора.

### Як виконувати домашнє завдання

Рекомендований підхід до роботи:
- уважно ознайомитися з умовою;
- обрати підхід до розв’язання;
- реалізувати рішення у директорії homework/;
- перевірити рішення, запустивши тести;
- за потреби звернутися до тестів під час вирішення задачі, використовуючи TDD як підказку до розвʼязання;
- поставити питання лектору, якщо щось залишається неочевидним;
- профілювати код і досліджувати його поведінку.

Під час реалізації намагайтеся інтерпретувати виконання коду в голові — що відбувається з пам’яттю, потоками, процесами
тощо (залежно від теми завдання).
Окремо рекомендується:
- свідомо обирати підхід перед вибором інструменту;
- аналізувати внутрішні механізми виконання;
- експериментувати з профілюванням і вимірюваннями.

У директорії homework_answers/ містяться запропоновані лектором рішення:
- використовується для порівняння реалізацій і підходів;
- публікується із затримкою (відповіді на попереднє ДЗ з’являються разом із наступним).

## Рев’ю і зворотний зв’язок

Після виконання домашньої роботи ви пушите зміни у свій `origin`. Лектор переглядає код, залишає коментарі і дає фідбек.
Відповіді на домашні завдання публікуються із затримкою навмисно — щоб зберегти простір для самостійної роботи і
порівняння свого підходу з референсом.

## Філософія курсу

Курс фокусується на **свідомому** технічному виборі. Ми розбираємо, як працюють системи під капотом, чому певні рішення
виграють або програють в конкретному контексті, і як аналізувати компроміси в продуктивності, складності та підтримуваності.
Сильного сучасного інженера визначають не бібліотеки у CV, а здатність ухвалювати виважені технічні рішення під
обмеженнями часу, бюджету, інфраструктури та команди. Саме це — головна мета курсу.
